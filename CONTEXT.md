# AI Vibe Coding Course — Context

หลักสูตรสอนคนที่**ไม่มีพื้นฐานการเขียนโปรแกรม**ให้สร้างเว็บแอปได้ด้วย AI (Vibe Coding)
โดยเชื่อมโยงให้เข้าใจแนวคิดเทคนิค (frontend / backend / database / SDLC / requirement / UX-UI)
ในภาษาที่คนไม่มีศัพท์เทคนิคเข้าใจได้ เครื่องมือหลัก: Antigravity 2.0 + VS Code + Gemini

## Language

**Vibe Coding**:
การบอกสิ่งที่อยากได้เป็นภาษาคนแล้วให้ AI เขียนโค้ดจริงให้ ผู้เรียนเป็น "ผู้กำกับ AI" ไม่ใช่ผู้พิมพ์โค้ด — แต่ผลลัพธ์คือโค้ดจริงที่ deploy ได้
_Avoid_: No-code (ใช้ได้เฉพาะเป็นคำการตลาด "สร้างได้โดยไม่ต้องเขียนเอง"; ในเนื้อหาห้ามใช้แทน Vibe Coding เพราะ no-code แท้คือเครื่องมือลากวางที่ซ่อนโค้ดไว้หมด)

**สะพานเชื่อมแนวคิด (Concept Bridge)**:
หลักการสอนของคอร์สนี้ — อธิบายแนวคิดเทคนิคผ่านการเปรียบเทียบกับสิ่งที่คนทั่วไปคุ้นเคย (เช่น HTML=โครงบ้าน, CSS=การตกแต่ง, JS=ระบบไฟฟ้า) เป้าหมายคือ "เข้าใจพอจะสั่งและตรวจงาน AI ได้" ไม่ใช่ "ท่องจำเพื่อเขียนเอง"

**เดโมกลาง (Central Demo)**:
โปรเจกต์ตัวอย่างตัวเดียวที่ผู้สอนนำสร้างตลอดคอร์ส ต่อยอดจาก "Bean There" (ร้านกาแฟ) เดิม → ยกระดับเป็นระบบรับจอง/สั่งล่วงหน้า เพื่อบังคับให้แตะครบทุกชั้น (requirement → UX → frontend → database → deploy)

**โปรเจกต์ของตัวเอง (Own Project)**:
ไอเดียจริงที่ผู้เรียนกลุ่ม A สร้างคู่ขนานไปกับเดโมกลาง ทุก hands-on ให้ลงมือกับของตัวเองด้วย กลุ่ม C ไม่ต้องมี ทำตามเดโมกลางพอ
_Avoid_: capstone, assignment (เลี่ยงคำที่ทำให้รู้สึกเป็นการบ้าน — มันคือของจริงที่เอาไปใช้ได้)

**พรอมต์ grill-me (Grill-Me Prompt)**:
เทคนิคเด่นของคอร์ส — พรอมต์ที่สั่งให้ AI สัมภาษณ์ผู้เรียนทีละคำถามเพื่อขุด requirement จนผู้เรียนพิมพ์ "หยุด" แล้วสรุปเป็น SRS ฉบับย่อ (ครอบคลุม: ใคร-ทำอะไร-เพื่ออะไร / ต้องมี-น่ามี / ข้อมูลที่ต้องเก็บเป็นตาราง). สะท้อนวิธี grilling ที่ใช้ออกแบบคอร์สนี้เอง
_Avoid_: interview script, questionnaire

**SRS (เอกสารสรุปความต้องการ)**:
ผลลัพธ์ของขั้น Requirement — เอกสารฉบับย่อที่สรุป ผู้ใช้ / ฟีเจอร์ต้องมี-น่ามี / ข้อมูลที่ต้องเก็บ. สอนเป็น "เฉลยท้าย" (ได้เอกสารก่อน รู้ชื่อทีหลัง). ตาราง "ข้อมูลที่ต้องเก็บ" ส่งต่อตรงเข้าโมดูล database

**Flow การใช้งาน (User Flow)**:
ผลลัพธ์ชิ้นแรกของขั้น UX/UI — เส้นทางที่ผู้ใช้เดินผ่านหน้าจอต่าง ๆ (เช่น เปิดเมนู → เลือกเวลา → กรอกข้อมูล → ยืนยัน → จองสำเร็จ). ตอบคำถาม "มีกี่หน้าจอ อะไรบ้าง"

**Wireframe (ภาพร่างหน้าจอ)**:
ภาพร่างหยาบ (กล่อง ๆ ไม่ใส่สี) ของหน้าจอหลัก 2–3 หน้า วาดบนกระดาษ → ถ่ายรูป → ส่งให้ AI อ่านและสร้างต่อ (หรือให้ Gemini generate mockup). คอร์สนี้สอนเฉพาะ "โครงสร้างการใช้งาน" — ความสวย (สี/ฟอนต์/UI visual) ปล่อยให้ AI+Gemini ไม่สอนทฤษฎีสี
_Avoid_: mockup, prototype (สงวนคำ mockup ไว้เรียกภาพสวยที่ AI สร้าง ไม่ใช่ภาพร่างมือ)

**เส้นทาง 6 ขั้น (The 6-Step Journey)**:
แกนเล่าเรื่องของทั้งคอร์ส = ลำดับธรรมชาติของการสร้างของจริง: 💡 Idea → ❓ Requirement → 🎨 UX/UI → 🛠️ Build → ✅ Test → 🚀 Deploy. ใช้แทนการสอน SDLC เป็นทฤษฎี — ผู้เรียน "เดินจริง" ตามขั้น แล้วค่อยเฉลยท้ายว่ามืออาชีพเรียก SDLC/Agile. จบที่ Deploy (Maintain เป็นแค่หมายเหตุสั้นท้ายขั้น Deploy ไม่ใช่ขั้นแยก)
_Avoid_: Waterfall, Spiral, phase (เลี่ยงศัพท์โมเดลทางการในเนื้อหาหลัก)

**ทดสอบแบบผู้ใช้จริง (User Acceptance Testing — ย่อ)**:
ขั้น Test ฉบับเบา 4 อย่าง: เดินตาม flow / กางเช็กลิสต์ "ต้องมี" จาก SRS / ลองทำผิด ๆ (edge case เบื้องต้น) / เจอ error แล้ววางให้ AI แก้. ปิด loop กลับไป requirement. ไม่แตะ unit test / automated test
_Avoid_: unit test, test case, QA
