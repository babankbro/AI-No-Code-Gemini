# M05 — เก็บ Requirement ให้ชัด 🆕

> 🏠 self-study + 🛠️ ลงมือกับโปรเจกต์จริง · ขั้น ❓ Requirement

## ปัญหาที่แก้
กลุ่ม A มีไอเดียในหัว แต่แปลงเป็น "สิ่งที่ระบบต้องทำ" ไม่เป็น → โมดูลนี้ให้เครื่องมือเบา ๆ ที่ลงมือได้ใน 1 หน้า

## เครื่องมือเก็บ Requirement 3 ชิ้น
1. **ใคร–ทำอะไร–เพื่ออะไร**
   เช่น *"ลูกค้า อยากจองโต๊ะล่วงหน้า เพื่อไม่ต้องรอคิว"* (เฉลยท้าย: นี่คือ user story)
2. **ต้องมี vs น่ามี**
   แยก MVP ออกจากของฟุ่มเฟือย → กันสร้างเกินจนไม่จบใน build day (เฉลยท้าย: MoSCoW)
3. **ข้อมูลที่ต้องเก็บ**
   ลิสต์ว่าระบบต้องจำอะไร (ชื่อ, วันจอง, จำนวนคน...) → ป้อนตรงเข้า [[M09 Database คืออะไร & ใช้ยังไง|database]] เป็นคอลัมน์

## 🔗 เส้นเชื่อมสำคัญ
- **Requirement = Prompt:** 3 ชิ้นนี้แมปตรงเข้า [[M03 การเขียน Prompt|สูตร C-G-R-E-F]] (Context=ใคร, Goal=เพื่ออะไร, Requirements=ต้องมี/ข้อมูล)
- **Requirement → Database:** ตาราง "ข้อมูลที่ต้องเก็บ" = schema เริ่มต้น
- **Requirement → Test:** เช็กลิสต์ "ต้องมี" = เช็กลิสต์ [[M12 ทดสอบ & Deploy ขึ้นออนไลน์|UAT]] ปิด loop

## เครื่องมือเด่น: [[พรอมต์ grill-me เก็บ requirement]]
สั่งให้ AI **สัมภาษณ์ผู้เรียนทีละคำถาม** จนพิมพ์ "หยุด" แล้วสรุปเป็น **SRS** (ดูศัพท์ [[อภิธานศัพท์#SRS]])
→ สะท้อนวิธี grilling ที่ใช้ออกแบบคอร์สนี้เอง
